# blog-polish

ブログ執筆ワークフローの 第4段階（最終）。
`drafts/{slug}/article.md` を点検し、`articles/{YYYY-MM-DD}-{slug}.md` として完成させる。

このスキルでだけ、ファイル名に日付（YYYY-MM-DD）を付与する。

## 入力

- 引数: `{slug}`
- 読み込み: `drafts/{slug}/article.md`

## 進め方

1. 全文通読のサマリ
   - 読み終えた印象を 3〜4 行でユーザーに返す
   - 「この記事を読んだ人が一番持ち帰るもの」が明確かを確認

2. 観点別チェック（1観点ずつ提示、勝手に書き換えない）
   - 冗長表現: 「〜することができる → 〜できる」など、削れる箇所を列挙
   - 一文の長さ: 80字を超える文を抽出し、分割案を提示
   - 用語ブレ: 同じ概念に複数の表記がないか（例: 「LLM」「大規模言語モデル」「AI」の使い分け）
   - 断定の根拠: 「〜である」「最も〜だ」と断じている箇所に、根拠を聞く
   - AI 臭: 「〜を活用することで」「〜と言えるでしょう」など、本人の声に置き換えるべき定型句を指摘
   - 専門用語: 初出で補足があるか
   - 各観点で修正案は出すが、適用するかは必ずユーザー判断

3. frontmatter の最終化
   - `title`, `summary`, `tags` を確定
   - `status: published`
   - `updated` を今日に
   - `created` は ideate 時の日付を維持

4. 公開（ファイル移動）
   - 公開日を確認（今日でよいか、別日に予約するか）
   - `articles/{YYYY-MM-DD}-{slug}.md` として書き出す
   - 元の `drafts/{slug}/` は消さない（後で経緯を振り返る資料）

5. コミット案内
   - `docs: add article "{title}"` 等、Conventional Commits の `docs:` プレフィックスを推奨
   - コミット自体はユーザーの明示的な指示があるまで実行しない

## 重要原則

- 推敲は指摘 → 本人判断 → 反映の往復。AI が黙って書き換えると本人の声が消える
- 「もっと良くなる」可能性は無限にある。完璧主義で公開を遅らせない判断を優先
- 一次体験・具体例を削らない。冗長表現の削減と混同しない

## 次のステップ

完了後、コミット & push（`git push origin main` 等）で公開できる旨を伝える。
